WordPress and web accessibility

When I joined the Global Change Program, their Sea Level Sensors website (sealevelsensors.org) was a static site generated through Jekyll. The Jekyll infrastructure was used to take advantage of the Markdown post feature, but the team wanted to lower the barrier of entry for post writing. This was part of a larger push to make our program’s various websites open to team members who did not have web development experience and simply wanted to edit a page or make a post. At that time, the one web developer was the only one who knew how to edit the Markdown, generate the static pages, and update the website files (then hosted through GitHub Pages) and therefore had to always be present as a mediator between the technology and the rest of the team.

My responsibilities were as follows:

  • Learn Jekyll and understand existing website code
  • Build a custom WordPress theme based on the existing styles
  • Migrate old website content onto new WordPress site
  • Publish WordPress site through Plesk server

As the only web developer on the communications team, I was responsible for all of the aforementioned items. To that list I also added a goal of my own: to improve the accessibility of the website. As I was not familiar with the many dimensions to web accessibility, the goal began as a rather broad one.

One challenge was my limited understanding of the breadth and depth of different accessibility practices. As a part of a public university, the web presence of the Global Change Program had implemented some accessibility practices, such as the consideration of color contrast, the inclusion of alt text, and the use of accessibility checkers (such as that in the Chrome inspect tool). Since I was responsible for the ground-up implementation of the WordPress site, however, I wanted to consider accessibility at all stages of the front end design and implementation process. In order to address this challenge, I turned to a variety of resources for self-study, many of which I use in front end projects to this day. For an understanding of design and user experience, resources such as the WCAG were essential, though in the early stages of my self-study I found the amount of information in the WCAG to be a bit overwhelming to read. For an understanding of implementation, resources such as the MDN Docs were helpful in explicating best practices for semantic HTML.

Through this particular website project, there were a number of accessibility considerations and practices to which I was newly exposed. One was a more holistic understanding of the user experience of people using keyboard navigation. This meant, for instance, that I had to consider whether interactable elements such as a collapsible menu would both allow for WordPress integration and for front end keyboard control. This also meant that the experience of keyboard navigation had to be not only functional but also clear and enjoyable in its own right: what kind of PHP templates allowed for a logical tabbing order so that users could open and close menu items without confusion? What kind of CSS styling allowed for a pleasant tabbing experience?
